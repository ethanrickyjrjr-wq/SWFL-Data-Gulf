## THE ONE THING TO UNDERSTAND

> **INVENTION IS CLAIM-SHAPED, NOT NUMBER-SHAPED.**

Seven workers built seven deliverables. **Four shipped a falsehood**, and every author had
opened the screenshot and signed off:

- *"$209/sq ft sits **below** the $213 median — and below the two recorded sales at $173 and
  $195."* — **$209 is ABOVE both.** The argument of a price-defense email, inverted.
- *"went under contract after **75 days**"* — the vendor's days-on-market is **null**.
- *"the seller reduced the price **before** a contract was reached"* — we hold a cut AMOUNT.
  No dates. **No ordering.**
- *"the gap is **widening**"* — from ONE data point. **A level is not a direction.**
- *"**five** of those six ZIPs"* — it was **four**.

**Not one contains an invented number.** Every figure was correctly sourced. What was
invented was **the claim drawn between correctly-sourced numbers**. A digit lint cannot see
any of it.

`lib/deliverable/claims.ts` closes it **structurally**: code computes the relation/count/
ordering, the narrator gets the RESULT as a settled sentence and **no raw set** — *it cannot
compare two numbers it was never given two of.* `auditClaims` is a fail-closed backstop.

**The gate has been beaten TWICE by adversarial verifiers, and both holes were ours.** It
matched only *spelled* counts (so `"All 6 comparable homes are recorded sales"` shipped clean
and false). It was blind to the model asserting things about the email's own layout.
**Assume there is a third.** Upgrade to entailment — check `claims_gate_entailment_upgrade`.

## WHAT IS BUILT AND LIVE (pushed to main)

**One recipe key.** A recipe's identity used to be its **prompt string**, and three registries
owned the concept with no shared authority. Now: a key (`lib/deliverable/recipes.ts`), carried
by every door as `?rkey=`, dispatched on by `authorDoc`. **All 12 builders registered** — every
one was *inert* before today (the dispatch table was still comments).

**The parity oracle** (`recipes.parity.test.ts`) fails the suite if any surface re-types a
prompt, if two doors offering one key disagree, if a pill's input contradicts its declared
subject, or if a chart-less recipe's prompt still promises a chart.

**The open-slot contract.** `stats`/`image`/`text` honor `emailRender`: an unsourced cell is a
"+ Add" affordance on the canvas and **absent** from the sent email. *(The old placeholder was
`"0"` — we were emailing zeros, i.e. invented numbers, to recipients.)*
⚠️ **`hero` and `signal` still do NOT honor it.**

**The seed/button collisions.** Four seed cards shared a name with a recipe button and built a
*different email*. Two of them **emailed coaching notes to real recipients** ("Say why this is
a good value now… a motivated seller means room to negotiate").

**The wrong-city guard.** `"I farm North Fort Myers"` resolved to **Fort Myers** and built a
confident, beautifully-cited email **about the wrong city**. The gazetteer can now say *"I do
not know this place"* → open slot, never a neighbour. **A gap is honest; a confidently wrong
subject is a lie that looks exactly like the truth.**

**The agent bio.** `agent_bio` was a **phantom field** — a textarea, a token, a renderer, and
**no column and no API entry**, so the save silently dropped it. Now real, and **it updates
itself**: the saved text carries `{{farm.home_value}}` tokens resolved at BUILD time with
source + as-of. Move the lake, the sent bio moves, the agent edited nothing.
`agent_profile_facts` stores one fact per row with a `source` column that is `NOT NULL` +
`CHECK` — **a fact with no provenance cannot be written**, so an invented credential is
structurally impossible.

**A design vocabulary.** The blocks had `{paddingY, sectionBg}`, `{kicker, value, label,
prose}`, `{value, label}`. No align, no emphasis, no band, no order. **The samples' design was
literally inexpressible** — which is why the built email never looked like the example, and it
was never a builder problem. Added: `BlockBase.align`, `HeroProps.ribbon` / `.order`,
`StatItem.emphasis`, `StatsProps.variant:"strip"` + footnote.

**One campaign chrome** (`lib/email/lifecycle-chrome.ts`). Seven lifecycle emails had **seven
different layouts** — each built by a different worker with its own idea of a grid, because
there was nothing to build onto. A subscriber walking Coming Soon → Sold would have received
seven emails that looked like seven different companies. Now every listing email is:

```
header · RIBBON · photo · hero(centred: ADDRESS over PRICE) · spec strip
       · [the recipe's own middle] · narrative · agent card · CTA · footer
```

A recipe changes the **ribbon word**, the numbers, its **middle** content and the **CTA**. It
does not get to invent a shape. **Brand is sticky** — `globalStyle`, header, agent card and
footer ride through from the canvas untouched. The chrome is the SHAPE; the brand is the SKIN.
Enforced by `lib/deliverable/campaign-coherence.test.ts`.

## WHAT IS NOT DONE

**The showcase samples still sell fiction.** *A sample is a sample* — the house need not exist
(operator, correctly). What matters is that **the layout is buildable with real data for a
typed address**, and it now is. **BUT:** Under Contract's sample has a hero slot for
**days-to-contract**, which **no vendor sells**. That layout has a permanent hole. Re-shoot the
samples from the real builder. Checks: `under_contract_showcase_premise_unsourceable`.

**The social path has NO no-invention gate at all.** Its sourcing rules are *prose in a
prompt*; `stat.value` is a free-text field the model writes. Same sourced lake feed as email,
**ungated on the way out**. `claims.ts` was built to be liftable onto it. Check:
`social_path_has_no_no_invention_gate`.

**The agent profile's AI drafter + growth loop** (pieces 2-UI and 3). See its handoff.

**Number hierarchy beyond the listing strip.** `StatItem.emphasis` now exists; the other 27
templates and the area recipes have not been given an order of importance. Check:
`stat_number_visual_hierarchy`.

**`sphere-weekly`** was refuted on its structural guarantee and needs one more pass.

**Engine debt** — three render engines disagree; `compile-grid` ghost column;
`upsertChartBlock` drops `layout`; the chart caption clips the as-of date off the PNG; the
shared chart producer binds YoY not MoM. All in the follow-ups doc.

## HOW TO WORK ON THIS WITHOUT REPEATING THE MISTAKES

1. **Build through the REAL path** — `authorDoc({prompt, rawDoc, recipeKey})` with **NO
   scope**. That is the Lab door, and it is the one that was broken.
2. **LOOK at the rendered artifact.** Grepping HTML is not looking at it.
3. **RE-DERIVE EVERY COMPARISON WITH ARITHMETIC.** Do not read it — compute it. *"$209 is
   below $195"* reads fine and is false.
4. **Run the real row grouper + `compileGrid`.** Assert every row's widths sum ≤ 12. **A
   picture that looks right in Chromium is not proof for an email** — a 1800px 3-column
   Outlook table passed a browser screenshot.
5. **A spec that names a figure no lane holds is an instruction to lie, and the model will
   obey it.** Fix the spec, not the model.
6. `authorDoc` **logs loudly** when a builder's doc fails schema and falls back to the generic
   author. If you see that, **what rendered was not your recipe.**
7. **Never `mock.module` a whole module** — spread the real one, or you silently strip exports
   and break innocent files.
