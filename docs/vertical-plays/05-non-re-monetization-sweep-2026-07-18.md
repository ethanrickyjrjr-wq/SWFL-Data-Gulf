## 1. SIRS / Milestone Compliance Radar (Lee + Collier)

**One-line pitch:** A monitoring feed that cross-references the dbpr_sirs_submissions table already sitting in the lake (verified live: 1,358 Lee+Collier association filings) against leepa_parcels/collier_parcels (actual_year_built, use_code, unit count) to flag condo/multi-unit buildings old enough to owe a SIRS or milestone inspection that have NO matching filing on record. A compliance-GAP finder, not a one-by-one lookup of what DBPR already publishes. Storm history supplies the 'why now.' Sold to CAM firms (subscription), board members/condo buyers (one-off diligence reports), and law/engineering firms (lead licensing).

**Why it ranked here:** Highest-value single find in the sweep: it resolves a real conflict in our own docs. STEADY-PAINS Tier 4 calls HOA/condo reserves a PERMANENT gap; this proves we actually hold compliance-FILING records (not reserve dollars/percentages), so the product is honest as a deadline/gap tool. Real, dated legislative pain (HB 913 layering CAM-verification + SIRS/milestone deadlines) and an unprompted in-buyer AI wishlist. Near-zero new build (a join across tables already ingested). The only honest ceiling: no reserve-funding %, so it must never claim reserve adequacy.

**Source domain:** HOA / condo association managers (CAMs)


### Implementation plan: Condo Compliance-Gap Radar (Lee + Collier): SIRS/Milestone Non-Filer Discovery Built on the Parcel-vs-Filing Join

Extends the existing condo-sirs-swfl brain (a positive-signal-only registry of 1,358 confirmed SWFL SIRS filings) by joining it against parcel_subdivision (a table already covering both Lee and Collier with year-built/unit-count/property-type) to build the missing baseline of condo/co-op buildings that plausibly owe a filing and have none on record. Verified live against the DBPR's own primary source that a large share of the true candidate universe already has a PAST-DUE statutory deadline as of today, sharpening the 'why now.' A live competitor, CondoClock ($399/mo/firm), already tracks statutory deadlines for buildings a firm already manages but does not do cross-source discovery of unwatched buildings and does not cover Lee County -- that discovery gap is the real wedge. The join itself is cheap; a defensible, non-defamatory product is not -- no table in the lake carries building story count (the law's actual 3+-story trigger), so the age/type/unit-count filter is a proxy that requires a human-review gate before any external claim, and DBPR's own association-name field is messy enough that naive matching risks false 'non-filer' claims against buildings that filed under a different name.

**One-line pitch:** Given a Lee/Collier subject address, return permit history (including permits pulled-but-never-closed), sale history, and storm/flood exposure alongside the county parcel's own GLA/year-built — positioned as independent cross-reference data (appraisers distrust BOTH MLS and assessor GLA), with the permit/renovation + storm layer as the defensible core neither MLS nor the assessor tracks together. A UAD-3.6 field-assist variant pre-populates the new mandatory form's structured fields ahead of the 11/02/2026 compliance deadline. Subscription ($40-80/mo/seat) or a licensing feed into ClickFORMS/TOTAL/ACI.

**Why it ranked here:** Best reach-to-reuse ratio on the board. Reach is genuinely SOLVED: DBPR publishes free bulk-CSV rosters for BOTH Home Inspectors (lic04home.csv) and the Real Estate Appraisal Board — same file shape our outreach engine already scrapes for boards 06/08 (mail/web channel, no email, matching the home-services precedent). Real, quantified pain (MLS data 'garbage,' assessor-GLA distrust, aging workforce) plus a hard dated regulatory deadline as a timing moat — only 3% of surveyed appraisers had done a UAD 3.6 job, 58% untrained.

**Source domain:** Property inspectors and appraisers


### Implementation plan: Appraiser & Inspector Subject Cross-Check (Permit / Storm / Flood / Assessor-GLA Discrepancy) -- UAD 3.6 Field-Assist Wedge

Live-verified plan for a Lee+Collier 'Subject Cross-Check' report that bundles FDOR parcel facts (assessor GLA/year-built/sale history), permit history (including a not-yet-built open/never-finaled flag), and storm+flood exposure into one independent artifact for appraisers and home inspectors -- with a UAD 3.6 field-assist layout timed to the real, sourced Nov 2, 2026 mandatory-compliance deadline. The honest headline finding from this session's research: the 'permits nobody tracks for appraisers' framing in the original idea is WRONG on its own terms -- BuildFAX (a Verisk business) already delivers permit history natively inside ClickFORMS today, verified live via Bradford Technologies' own support docs. The defensible differentiator is not permits alone; it's the BUNDLE (permit+storm+flood+parcel-GLA-discrepancy in one artifact, which BuildFAX does not do), Lee+Collier hyperlocal depth (102/120 FDOR fields), the dated UAD 3.6 wedge, and price -- and the plan is written around that corrected framing. A second hard rule this research surfaced: GLA is the one number appraisers distrust from BOTH the MLS and the assessor (real quoted evidence) -- so the product must never auto-fill GLA as authoritative, only ever surface it as a sourced, dated discrepancy flag, enforced structurally by the existing claim-gate. ## 9. Grounding Gate — licensable never-invent / citation validator API

**One-line pitch:** Extract our own production lint battery (spec-validator, facts-only-lint, inference-bait-lint, smoothing-lint, and the gateNarrative no-invention gate) into a standalone, domain-agnostic API/SDK sold under a separate developer-tool brand. A builder posts an LLM output plus its source context; the service returns pass/fail, the list of unsourced/unsupported claims, and a confidence score. Usage-based pricing + a team subscription tier, priced under the incumbents.

**Why it ranked here:** The purest expression of the operator's 'reuse an engine for a new problem' framing and the board's 'engine is the asset, storefront disposable' rule — this code already runs in production. Demand is real and large: HN-scale threads (a 1,713-point 'losing trust in AI agents' post), builders hand-rebuilding our exact facts/rules/citation-gate architecture from scratch and getting upvoted for guessing at it, and funded competitors (Patronus, Parsewise) productizing adjacent pieces. Ranked here, not top-5, on two honest caveats the sweep flags: it has ZERO SWFL data moat and requires a whole new developer-GTM muscle against funded rivals — high ceiling, off our core axis.

**Source domain:** Indie SaaS founders / AI-tool builders


### Implementation plan: Grounding Gate — a claim-level LLM output validator API, sold under its own developer-tool brand (not SWFL Data Gulf)

Real code exists to reuse, but the idea as pitched overstates how much is a "repackage." We run three real, in-production defenses against fabricated LLM output: (1) refinery/validate/{facts-only,inference-bait,smoothing}-lint.mts — regex-based lints on our own brain markdown, scanning a proprietary ```reference fence for injection phrasing, hedge words, and one platform-specific cross-brain causal check; (2) lib/deliverable/narrative-lint.ts's numeric verbatim-anchoring (extractNumbers/normalizeNumber/anchorsExactly) — a genuinely domain-agnostic "does every digit in this prose appear verbatim in the source snapshot" checker, the single most reusable piece found; (3) lib/deliverable/claims.ts's auditClaims/SettledClaim — a fail-closed detector for unsupported comparative/trajectory/count/sequence/spatial/motive claim SHAPES, which is a genuinely good design idea but its regexes (HOME_FEATURES: dock/lanai/seawall/granite; MARKET_EVENT: contract/listing/pending; MOTIVE: "seller is motivated") are ~90% real-estate vocabulary, not domain-agnostic patterns. Crucially, all three mechanisms work today because THIS platform controls the generation step — the narrator is never shown a raw number pair, only pre-computed "settled" sentences, so a closed-world regex check suffices. A third-party API that receives someone else's already-generated output plus arbitrary source context, after the fact, cannot assume that closed world — it has to do real claim-extraction-and-entailment checking, which none of our code does. That is the same hard problem Cleanlab's TLM (0.91 AUROC benchmark, just acquired by Handshake AI in 2026), Patronus, Galileo, and RAGAS Faithfulness already compete on. So the honest plan is a two-layer product: Layer A is a genuine, fast, near-free extraction of our domain-agnostized numeric-anchor + hedge-word + injection-pattern checks (real reuse, days not weeks); Layer B is net-new — an LLM-as-judge entailment engine built on the platform's existing forced-tool-call/spend-guard infrastructure, using the claim-shape taxonomy as a prompted rubric instead of fixed regexes. Live competitor research (crawl4ai, 07/18/2026) found the category has real paying demand and an exit (Patronus self-serve at $10-20/1k calls; Galileo free-to-$100/mo; Guardrails AI retreated from a raw API to OSS+enterprise sales; Cleanlab TLM — the closest 1:1 match — was just acquired by Handshake AI and is now demo-gated; Parsewise is YC-backed but scoped to insurance/risk documents, enterprise-only pricing; a crowded indie cluster of academic-citation checkers proves builders rebuild narrow slices of this fast) — the real wedge is transparent, claim-level (not just a scalar score), usage-based self-serve pricing in a moment when the nearest 1:1 rival just exited to M&A, not "nobody does this."


**Build steps:**

1. 1. Scope the MVP honestly in a spec doc BEFORE writing code (per RULE 3.5 brainstorm-first + RULE 0.5 probe-first): two layers, not one extraction. State explicitly that Layer B is new build, not lifted, so the plan can't quietly regress into overclaiming.
2. 2. Extract Layer A: fork (not import, to avoid coupling a public product to this private repo) the three domain-agnostic primitives — extractNumbers/normalizeNumber/anchorsExactly/buildAnchorSet from lib/deliverable/narrative-lint.ts, the SMOOTHING_TOKENS list from refinery/lib/smoothing-tokens.mts (kept as a caller-overridable default, not a fixed constant), and the second-person/injection PATTERNS from refinery/validate/facts-only-lint.mts (drop the hardcoded SWFL 'local context [...]:' exemption regex — replace with a caller-supplied citation-marker config). Generalize the input contract from this repo's ```reference-fenced markdown to plain {output: string, context: string | string[]}.
3. 3. Re-derive the claim-shape taxonomy from lib/deliverable/claims.ts (comparative / trajectory / count / sequence / spatial / motive / positional-in-document) as a DOMAIN-AGNOSTIC CONCEPT — keep the taxonomy (it's a real, differentiated design idea vs. competitors' single opaque score), but do not port the HOME_FEATURES/MARKET_EVENT/MOTIVE word lists verbatim; they are real-estate vocabulary and would misfire on legal, medical, or financial text.
4. 4. Build Layer B: one forced-tool Claude call per validation request (reuse the harness pattern in refinery/agents/anthropic.mts / synthesis-agent.mts), prompted with the claim-shape taxonomy as a rubric: 'given ONLY this source context, list every claim in this output that is a comparison/trajectory/count/sequence/spatial-relation/motive-attribution not directly supported by the context, and every numeral not present verbatim.' This is the net-new, expensive-to-build piece — nothing in the repo does open-world entailment checking today.
5. 5. Port ingest/lib/api_usage.py's RunBudget pattern (hard per-request/per-day spend ceiling, same $/MTok rate table) to TypeScript as the internal cost guard on every Layer-B call — this protects margin on a product whose cost basis is a real Claude API call, not a free regex.
6. 6. Merge Layer A + Layer B into one response contract: {pass: boolean, unsupportedClaims: [{text, kind, reason}], confidence: 0-1, layerUsed}. Ship Layer A alone as a cheap/free-tier fast-path; gate Layer B behind the paid tier.
7. 7. Benchmark the combined system against a NAMED public hallucination-eval dataset (e.g. the same class of RAG benchmarks Cleanlab published against — CovidQA/DROP/FinanceBench/PubmedQA-style sets) before publishing any accuracy claim.
