# Lane 08 — Backtest / Grading / Flywheel (the moat engine)

**Health: mostly-ok (engine sound, loop stalled by plumbing, not by math).**

The pure decision/grading math is genuinely careful: point-in-time look-ahead guards
(`pitInitial` gated on `realtime_start <= asOf`), initial-vintage discipline, a real
persistence-null skill baseline with a shared denominator, per-slug polarity that is
never inherited, and a write-once atomic grade RPC. The honest first read (system 42.0%
vs persistence 48.6% → **lift −6.5pp**) is correctly reported as "does not beat naive,"
not buried. The problems are at the seams, not the kernel: the retrodicted corpus is
generated by a **manual, un-scheduled, snapshot-pinned** one-shot (it never refreshes);
the **LIVE** loop has banked **zero** outcomes (`outcomes`=0) and the backtestable
universe is **one series (LAUS)** because everything else is revision-dirty or has no
retained vintages. The flywheel can in principle close, but today it is a built engine
with no fuel flowing through it.

---

## [HIGH] The retrodicted corpus is generated by a manual one-shot — no workflow runs it, and the snapshot is pinned

**Location:** `refinery/tools/flywheel-backtest.mts:112` (`DEFAULT_SNAPSHOT = "2026-06"`), no `.github/workflows/*` invokes it (confirmed: grep of `.github` for `flywheel-backtest` → 0 hits; only `grade-predictions.yml` exists for the live loop).

**Detail:** `flywheel-backtest.mts` is the only producer of `backtest_grades`, and it is a
hand-run CLI. The ALFRED LAUS pipeline (`fred-laus-alfred-monthly.yml`) refreshes the
Tier-1 parquet monthly, but the harness reads a **hard-pinned** `2026-06.parquet` snapshot
unless `--snapshot` is passed manually. Net effect: the 144 retrodicted grades written on
2026-06-07 are frozen. New ALFRED vintages (which extend the as-of grid and would add
fresh non-overlapping calls every quarter) never enter the corpus, so Panes 3/4 of The
Glass calibrate against a static, aging snapshot. The whole point of "the future is the
past now" decays the moment the snapshot stops advancing.

**Fix:** Add a low-frequency workflow (monthly, downstream of `fred-laus-alfred-monthly`)
that runs `flywheel-backtest.mts` with `--snapshot` defaulting to the **latest** uploaded
parquet (or resolve "newest" from S3 rather than a constant). Keep it idempotent (it
already upserts on the natural key). Gate writes behind the same service-role secret the
grader uses.

**Model:** opus — touches the retrodicted-corpus refresh contract + snapshot-resolution
semantics (a wrong "latest" choice silently re-grades on revised data, the exact look-ahead
the module is built to avoid).

## [HIGH] LIVE forward loop has banked zero outcomes — and the only live-gradeable yield is leaf sign-slugs that hadn't refreshed since before §6-A wiring

**Location:** live DB reality from `SESSION_LOG.md:975` and `:1464` (40 master preds, 11 carry claims, 29 legacy `pending` husks, `outcomes`=0); `refinery/grade/grade-predictions.mts:271` (`selectDue`), `refinery/lib/predictions-log.mts:253` (`deriveSlugPredictions`); `refinery/stages/4-output.mts:697` (`logSlugPredictions`).

**Detail:** The grading loop is closeable in principle, but in practice it has banked
nothing. Master's synthesized top-line is `mixed` most of the time (the correct, honest
behavior — see `voteDirection` < 0.60 agreement → `predicted_direction = null` →
ungradeable), so the synthesis tier produces almost no gradeable rows. §6-A (per-slug leaf
sign-direction sub-calls) is the actual yield multiplier, but `SESSION_LOG.md:975` records
that the 12 sign-slug leaf brains last refined 06-03/06-06 — *before* the §6-A wiring
landed (`a4f5383`, 06-08) — so `logSlugPredictions` had banked 0 rows at audit time. Even
once rows land, the FIRST outcome cannot resolve until a full window (90–395 days depending
on category) has elapsed AND a later `metric_observations` row for that slug exists at/after
`window_end_date`. So "outcomes=0" is partly just time, but the missing-refresh means the
clock had not even started. There is no monitoring that the live queue is actually filling.

**Fix:** (1) Confirm the daily rebuild now refreshes the sign-slug leaves and that
`logSlugPredictions` is inserting `kind='slug'` rows (one-line DB check:
`SELECT prediction_kind, grade_status, count(*) FROM predictions GROUP BY 1,2`). (2) Add a
freshness/health signal (a `checks` row or an ops tile) that flags when 0 gradeable
predictions have been opened in N days — the loop silently producing nothing is the failure
mode that already bit it once. (3) Filter the 29 legacy `pending` husks out of any
Pane-2/`fetchOpenCalls` read (`grade_status IN ('gradeable','ungradeable')`), per
`SESSION_LOG.md:1464`.

**Model:** opus — requires reasoning about loop-closure timing across capture/observe/grade
and the honesty constraint that mixed must stay ungradeable (no manufacturing bets).

## [HIGH] Backtestable universe is a single data family (LAUS) — the moat's effective N is bounded by one revised aggregate

**Location:** `refinery/tools/flywheel-backtest.mts:59-70` (`BACKTESTABLE` = only `laus_lee_*` + `laus_collier_*`); 11 "backtest-clean" slugs declared in `refinery/tools/grade-config-sweep.mts:54-66` but none are wired into the harness; `EXCLUDED` log `:72-89`.

**Detail:** The sweep advertises 11 vintage-clean gradeable slugs (3 SBA survival/chargeoff,
7 TDT hospitality, 1 LeePA velocity z). The retrodiction harness uses **none** of them — it
runs over the two LAUS unemployment series only, and LAUS is explicitly the *dirtiest*
forward slug (BLS benchmark revisions restate history; `SESSION_LOG.md:2193`). The 11
"clean" slugs are clean precisely because they come from immutable individual-record tables
(SBA loan outcomes, TDT remittances, LeePA deeds) — exactly the series that would give an
honest grid without ALFRED-style vintage reconstruction. But TDT is "fixture-only / self-
ingest pending" and LeePA "keeps only the latest sale per parcel" (a real grid-honesty
defect, correctly excluded). So the corpus's effective independent N is tiny: 144 calls but
only **2 series / 1 phenomenon** (the persistence skill metric's `n_families` is the honest
N, and it is ~2). A −6.5pp lift on one autocorrelated macro series is a weak basis for "the
call logic needs work."

**Fix:** Land the SBA and TDT clean series as honest grids (SBA outcomes are immutable —
no vintage reconstruction needed; TDT needs the self-ingest that is already tracked). Each
adds an independent family to the skill denominator. Until then, caption every retrodicted
number with its family count, not just call count (the harness already computes
`n_families` — surface it in The Glass, not just the console).

**Model:** opus — it is a corpus-design/data-acquisition judgment (which series are
grid-honest) intertwined with the vintage-policy invariant.

## [MEDIUM] `signalConfidence` is a deadband-derived proxy, but it is fed into a calibration pane that reads as "system is 80–100% confident, 41.5% right"

**Location:** `refinery/lib/backtest/grid.mts:145-164` (`signalConfidence`), consumed by `refinery/tools/flywheel-backtest.mts:183` (`calibration`) and the `glass_calibration` view (`SESSION_LOG.md:1403` — "80–100% stated → 41.5% actual").

**Detail:** The retrodicted call has no LLM-stated confidence, so `signalConfidence`
manufactures one from how far the as-of move clears its deadband, saturating toward 0.95.
On a trending macro series most moves clear the deadband comfortably, so nearly every call
lands in the top confidence band — which is then read by the calibration pane as systematic
**overconfidence** (high stated confidence, ~chance hit-rate). That is an artifact of the
proxy's construction (signal strength ≠ probability of being right on a mean-reverting/
persistent series), not a real confidence-calibration finding. The module comment honestly
labels it a proxy "that tunes NOTHING," but downstream the number is plotted on a
calibration curve as if it were a probability, which invites a wrong conclusion ("our
confidence is miscalibrated") about a system that never emitted these confidences.

**Fix:** Either (a) exclude retrodicted `signalConfidence` from the calibration pane and
reserve that pane for LIVE LLM-stated confidences only, or (b) relabel the retrodicted pane
explicitly as "signal-strength buckets, not probability calibration" so no one reads
overconfidence into a deterministic proxy. The honesty guardrail already keeps retrodicted
numbers out of public claims; this is the same discipline applied to the internal read.

**Model:** sonnet — well-specified relabel/exclude once the decision (a vs b) is made; the
decision itself is a small judgment call.

## [MEDIUM] Skill baseline excludes neutral-observed targets entirely — a directional system that correctly predicts "no move" gets no credit, and the deadband choice silently sets the scored N

**Location:** `refinery/lib/backtest/skill-baseline.mts:111-131` (neutral target dropped), `grid.mts:238` (a neutral *call* is dropped as "no bet").

**Detail:** Two coupled honesty choices interact in a way worth flagging. (1) The system
never logs a neutral *call* (a within-deadband as-of move → `buildGradedCall` returns null).
(2) The scorer drops any target whose *observed* outcome is neutral. Together this means the
skill metric only ever scores directional-call-vs-directional-outcome pairs. That is
defensible (a directional bet vs an inconclusive outcome is genuinely unscoreable), but it
has a side effect: the **deadband epsilon** (`VALUE_TYPE_BUCKET`, e.g. 0.05 for ratios)
silently controls the scored N — a larger epsilon classifies more moves as neutral and
shrinks the denominator, a smaller one inflates it. The epsilon is a methodology constant
with no per-slug justification in most cases (it inherits from `value_type`), so the scored
N — and therefore the lift's statistical weight — is sensitive to a constant nobody
re-derived per metric. There is no sensitivity report.

**Fix:** Add a small sensitivity line to the harness output: scored N and lift at
0.5×/1×/2× the deadband, so a reviewer sees whether the −6.5pp result is stable or an
artifact of the epsilon choice. Keep the neutral-drop semantics (they are correct); just
make the denominator's dependence on epsilon visible.

**Model:** sonnet — additive, well-scoped reporting change; no invariant touched.

## [MEDIUM] `safety_property_crime_yoy_pct_*` is categorized `real-estate`, giving a property-crime metric a 180-day real-estate grade window

**Location:** `refinery/vocab/brain-vocabulary.json` — `safety_property_crime_yoy_pct_{lee,collier,swfl}` have `category: "real-estate"` (confirmed by full-vocab gradeable enumeration); window comes from `CATEGORY_WINDOW_DAYS["real-estate"] = 180` in `refinery/vocab/loader.mts:137`.

**Detail:** These are FDLE/FBI crime YoY change slugs (sign-basis, `lower_is_bullish`,
which is correct). But their category is `real-estate`, so they inherit the real-estate
180-day grade window. Crime data publishes on a different cadence (FDLE/CDE annual-ish,
`fdle-crime-quarterly.yml`), so the inherited window is a category-mislabel artifact, not a
cadence-grounded choice. Because polarity is slug-declared (correct), the *direction* grades
right; only the *window* (when the outcome is read) is governed by the wrong category. A
mis-sized window changes which future observation resolves the call — too short and it
grades against a not-yet-published vintage; too long and the call's relevance horizon is
overstated. This is a quiet correctness drift, not a crash.

**Fix:** Either re-categorize these to a crime/safety category with a window grounded in the
FDLE/CDE release cadence, or add explicit `grade.window_days` blocks on the three slugs so
they stop inheriting the real-estate default. (The polarity-is-slug-only design already
proves per-slug overrides are the sanctioned escape hatch.)

**Model:** sonnet — mechanical vocab edit once the correct cadence window is picked; verify
against the publisher's calendar.

**Web question:** What is the actual publication cadence/lag of FDLE UCR / FBI CDE NIBRS
property-crime annual figures for Florida counties (to set an honest grade window)?

## [LOW] Grader's `findObservation` is single-vintage / first-after-window — it cannot detect that the resolving observation is itself a later-revised value

**Location:** `refinery/grade/grade-predictions.mts:290-300` (`findObservation` → `metric_observations` `observed_at >= windowEndDate`, earliest), `pickEarliestObservation:146-159`.

**Detail:** The retrodiction harness is scrupulous about initial-vintage PIT discipline
(`pitInitial` never reads a later revision). The **live** grader has no equivalent guard:
`metric_observations` stores one value per `(slug, brain_id, observed_at)` with
`observed_at = refined_at`, and the grader takes the earliest row at/after the window close.
If the underlying source revised the metric between the prediction's capture and the window
close (LAUS benchmark revisions, ZHVI/ZORI re-writes), the grader grades against the revised
value with no flag — the same look-ahead the backtest module explicitly forbids, now on the
live path. For immutable-record slugs (SBA/TDT/LeePA) this is a non-issue; for the revised
aggregates it silently contaminates the live verdict.

**Fix:** For revision-prone slugs, either grade only on initial-vintage tables (the
`*_initial_vintage` family the harness uses) or stamp the observation's vintage and refuse
to grade against a value whose vintage post-dates the window close. At minimum, record on
the outcome whether the resolving observation was a first-print or a revision, so a later
audit can separate clean from contaminated live grades.

**Model:** opus — it is the live-side mirror of the PIT invariant; getting it wrong
re-introduces the exact bug the backtest layer was built to prevent.

## [LOW] No reconciliation test pins the SQL skill views to the TS scorer — they were reconciled once by hand

**Location:** `SESSION_LOG.md:1403` ("corrected `glass_skill_over_time` … reconciled EXACTLY to the TS scorer N=138 / 0.4203 / 0.4855 / −0.0652"); SQL views in `docs/sql/20260608_glass_views.sql` / `20260608_data_targets.sql` (`backtest_skill_by_slug` via `LAG`); TS scorer `refinery/lib/backtest/skill-baseline.mts`.

**Detail:** The persistence-null lift is computed twice — once in TS (`computeSkillScore`)
for the harness, once in SQL (`LAG`-based views) for The Glass / `data_targets`. They were
reconciled manually to identical numbers, but there is no test that re-asserts the SQL
window-function persistence null matches the TS positional shift (including the deliberate
"neutral prior counts as a persistence miss" edge and the non-first/non-neutral denominator).
A future tweak to either side (e.g. a different `PARTITION BY`/`ORDER BY`, or an epsilon
change) can silently diverge them, and the divergence would only surface as a confusing
"the view says X, the harness says Y" much later.

**Fix:** Add a golden-fixture reconciliation test: a fixed set of scored calls, the expected
`(system, persistence, lift, n)` from `computeSkillScore`, and the same set run through the
SQL view (or a TS re-implementation of the exact LAG semantics) asserting byte-equality.
Pin the neutral-prior and first-call edges explicitly.

**Model:** sonnet — additive test against an existing, already-reconciled contract.
