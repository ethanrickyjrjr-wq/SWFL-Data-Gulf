# Vintage Policy Revision Audit — 2026-06-04

> **Purpose:** Identify which gradeable slugs are point-in-time honest for backward backtest.
> Read-only. No schema changes. No cadence_registry edits. One deliverable.

---

## Methodology

1. `ingest/cadence_registry.yaml` — every registered source + pipeline lane
2. `refinery/vocab/brain-vocabulary.json` — all concepts with `grade.direction_polarity` (slug-level only; polarity is never inherited per `loader.mts` line 80)
3. `refinery/vocab/loader.mts` — `CATEGORY_WINDOW_DAYS` and `VALUE_TYPE_BUCKET` defaults
4. Source trace — `refinery/sources/*-source.mts` comments + cadence_registry table names

**Gradeable gate** (from `resolveGradeConfig`): slug must have explicit `grade.direction_polarity` ≠ "none" AND be in a category with a `CATEGORY_WINDOW_DAYS` entry (e.g. "regulatory" is absent → any news slug with a grade block still fails).

**Backtest-clean logic:** overwrite ≠ dirty. Individual records (deeds, loan outcomes, tax remittances, storm events, NFIP claims) are immutable. Only **revised aggregates** — sources that publish backward-restatements of historical period values — are dirty for backtest.

## LAUS NOTE

**`laus_lee_unemployment_rate` backtest status: DIRTY — NOT SAFE FOR BACKTEST**

This slug is simultaneously the most visible forward-flywheel slug and one of the least trustworthy for backward backtest. Reason:

- BLS LAUS publishes **preliminary** monthly estimates, then applies **annual spring benchmark revisions** that restate every prior month in the series back ~18 months
- A prediction made in March 2024 against the then-published Lee unemployment rate of X% may have been judged against a value that BLS later restated to Y% — the table currently holds Y%, not X%
- The pipeline (`bls_laus`, dlt, `data_lake.bls_laus`) does a normal upsert; there is no `as_of_date` column preserving the point-in-time preliminary value
- `vintage_policy_rec: append_asof` — each monthly pull should store `(series_id, period_date, value, pulled_at)` so the as-of-then preliminary value survives benchmarks
- Until append_asof is implemented, any backtest result using LAUS is measuring against a **revised** benchmark, not the information set available when the prediction was made
